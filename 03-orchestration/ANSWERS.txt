Q4: Run 5_web_research_agent.yaml with the default research topic about data orchestration trends.

In this flow, who decides when to use the web search tool?

* The agent autonomously decides based on the prompt and system message. You specify the GOAL, then the agent specifies how to achieve it. In the yaml file we can see different variables, such as systemMessage (which encourages searching), contentRetrievers and tools that are declared as the capabilities of the agent. But there is no explicit Search task in tasks: the only top level task is research_agent which is the manager to call Search function inside the agent's autonomous loop.

Q5: Examine 6_multi_agent_research.yaml and run it with the default company (kestra.io).

What is the role of the research agent in this multi-agent system?

* It serves as a tool for the main agent to gather web data. If you look at the reserach_agent, it is nested inside the main agent's loop. Kesstra explicitly exposes an Agent-As-A-Tool wrapper, which means "let one agent invoke another one the same way it would call any other tool".

Q6: Based on what you learned in this module, for production workflows requiring deterministic, repeatable results with strict compliance requirements (e.g., financial reporting, workflows in highly regulated industries), which approach is most appropriate?

* Use traditional task-based workflows for predictability and auditability. We are talking about predictable and repetitive solutions, that means that including an AI agent in the response could make mistaken and not predictable answers. Same inputs must produce same outputs in deterministic solutions, agents are probabilistic. That means the same agent with the same input can make two different answers for the same question, so, in this case, we need to use traditional task-based workflows.
